# LLM (ニューラル) 補正の調査メモ

作成日: 2026-07-20 (文脈補正フェーズ、Phase D)

本メモは roadmap.md の「優先度D: 長期の研究枠」にある azooKey の Zenzai 方式の
調査結果。文脈補正・文脈学習 (統計方式、docs/protocol.md の CONVCTX/LEARN2) は
実装済みだが、本メモの LLM 補正は**調査のみでコードは未着手**。

## Zenzai / mozkey の Zenz 方式の概要

- azooKey の Zenzai は、かな漢字変換を「読み → 表記」の条件付き生成として
  GPT-2 系の小型言語モデル (パラメータ数 0.3B 程度、GGUF 形式) で行う方式。
  Rust バインディングを介して llama.cpp でローカル推論する
- 統計変換 (ビタビ + 連接コスト) を完全に置き換えるのではなく、
  ドラフト (投機的デコーディング) として使い、LLM の出力と照合することで
  1トークンずつ全部を推論するより高速化する設計になっている
- mozkey (Mozc fork、本セッションの調査対象) は Zenzai をベースに、
  「Zenz ライブ補正」という形で統合している:
  - パイプライン: `mozc_server` → named pipe → `mozc_zenz_scorer.exe` (仲介) →
    `llama-server.exe` (localhost 推論)
  - 直前の確定済み左文脈をプロンプトに含め、助詞 (に/を/が) が同音漢字に
    負けないよう候補順位を補正する
  - 出力の妥当性検証 (空出力・長すぎる出力・安全性チェック) を経てから採用
  - ユーザ辞書語や ASCII/mixed-script 表記 (例:「Mozkey」) は保護し、
    LLM が勝手にカタカナへ上書きしないようにする
  - フィードバック学習: 採用/却下を TSV に記録するが、生の左文脈は保存せず
    「読み」「補正後の表記」「非可逆な context class (例: empty / japanese_only /
    mixed_japanese_ascii / sensitive_like)」のみを保存する。次の変換で
    このフィードバックスコアが候補コストの調整に使われる

## QuicklIME に組み込む場合のアーキテクチャ案

現行の文脈補正 (本フェーズで実装した CONVCTX/LEARN2、engine/src/convert.rs の
`resolve_context_id`) がそのまま LLM 側のプロンプト用の前文脈としても使える。

1. **プロセス構成**: エンジン (quicklime-engine.exe) に別スレッドで llama.cpp
   (llama-cpp-rs 等の Rust バインディング) を常駐させる。TSF 層からの接続や
   プロトコルは変えず、CONVCTX の処理内で追加の推論ステップを挟む
2. **応答方式**: LLM 推論は CPU でも数十〜数百ms かかるため、ライブ変換の
   毎打鍵パス (ConvertSegmentsLive、非ブロッキング前提) をブロックしないことが
   必須。案:
   - 通常の CONVCTX は現行の統計変換のみで即応答する (現状維持)
   - 別コマンド (例: `RERANK`) を新設し、TSF 側が変換確定前 (Space 押下後の
     候補表示中など) に非同期で investigate し、結果が間に合えば候補順に反映する
   - もしくは mozkey 方式のように、統計変換の N-best をドラフトとして LLM に
     渡し、投機的デコーディングで検証コストを抑える
3. **文脈の受け渡し**: 本フェーズで TSF 層に追加した `contextReading_`/
   `contextSurface_` (直前確定文節) をそのまま LLM プロンプトの左文脈として
   使える。ハイブリッド照合 (Phase C) の仕組みも流用できる
4. **フィードバック学習**: mozkey に倣い、生の文脈文字列は保存せず
   「読み・補正後表記・非可逆な context class」の組で記録する方式が
   プライバシー的に安全。既存の `LearningStore` の ctx_map とは別テーブルにする

## コスト見積り

- **モデルサイズ・配布**: 0.3B パラメータ級の GGUF でも量子化込みで数百MB。
  現行の QuicklIME はインストーラで Mozc 辞書 (数百MB) を同梱済みだが、
  さらにモデルを追加すると配布サイズが倍近くに膨らむ
- **常駐メモリ**: llama.cpp のコンテキストサイズ次第だが、常駐で数百MB〜1GB
  程度を追加で占有する。IME は常時起動が前提のプロセスなので、ユーザ環境
  (特に省メモリ機・古い機種) への影響が大きい
- **レイテンシ**: CPU 推論で1回の生成に数十〜数百ms。ライブ変換や毎打鍵の
  サジェストと同じ体感速度を求めると、GPU 前提かごく小さいモデルでないと
  厳しい可能性がある
- **保守コスト**: llama.cpp・モデルファイルのバージョン管理、Windows 環境での
  ビルド (CUDA/Vulkan 等のバックエンド選択)、ライセンス確認 (Zenzai の
  モデル配布条件) が追加で発生する

## 見送り理由と再検討の条件

- 現時点 (本フェーズ) では、統計方式の文脈補正 (CONVCTX) と文脈学習 (LEARN2)
  で「服を|着る」「橋を|渡る」のような文脈依存の同音異義語の多くはカバーできる
  見込みがあり、投資対効果の観点で LLM 補正は優先度を下げる
- 再検討の条件:
  - 統計方式 (連接コスト + 文脈学習) を運用してみて、なお誤変換が残る具体例が
    一定数溜まった場合 (「文脈補正では拾えないパターン集」を別途 memo.md 等に
    溜めておくとよい)
  - 十分に小さく高速な日本語かな漢字変換特化モデルが公開され、
    CPU 推論でもライブ変換の体感速度を損なわない目処が立った場合
  - ユーザがモデル配布サイズ・常駐メモリの増加を許容する場合
