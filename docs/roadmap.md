# QuicklIME 開発ロードマップ

作成日: 2026-07-11

## プロジェクト概要

- **QuicklIME**: Windows 用の自作日本語IME
- **目的**: 常用に耐える実用IMEを作ること（学習目的ではない）
- **入力方式**: 通常のローマ字入力。変換を工夫して様々なシーンでの高速入力を目指す

## アーキテクチャ

Mozc / Google日本語入力と同じ「薄いクライアント + 別プロセスエンジン」構成を採用する。

| 層 | 言語 | 形態 | 役割 |
|---|---|---|---|
| TSF層 | C++ | in-proc COM DLL | Windows に IME として登録、キー捕捉、composition 管理、候補描画 |
| 変換エンジン | Rust | 常駐別プロセス | ローマ字かな変換、かな漢字変換、学習 |
| (将来) 設定UI等 | 未定 | 別実行ファイル | 設定画面、ユーザ辞書管理 |

### この構成を選んだ理由

- TSF のテキストサービスは in-proc COM DLL であり、全アプリのプロセスにロードされるため、軽量・堅牢な C++ 以外の選択肢が実質ない
- エンジンを別プロセスにすることで:
  - 学習データを全アプリで共有できる
  - エンジンがクラッシュしてもホストアプリを巻き込まない
  - 32bit/64bit 問題がエンジン側では消える（DLL は両アーキテクチャ分必要）
  - エンジンの言語を自由に選べる → 品質重視で Rust を選択
- 検討した代替案: 全部C++（メモリ安全性とイテレーション速度で見送り）、C#エンジン（品質重視のため見送り。ただし設定UIには将来採用の可能性あり）

### フェーズ4: かな漢字変換エンジン本体 (Rust)

- [x] 辞書: Mozc OSS 辞書 (約129万エントリ) を流用。現状 HashMap 実装
      (共通接頭辞検索が性能ボトルネックになったら yada への置き換えを検討)
- [x] Viterbi による最小コスト経路探索 (連接行列対応)
- [x] 文節区切りと部分変換 (品詞ベースの文節化、←→で文節移動、文節別候補選択)
- [x] 学習 (確定履歴による候補順の調整)。%APPDATA%\QuicklIME\learning.tsv へ永続化
      (現状は「読み→最後に確定した表記」の単純方式。頻度ベース等への改良余地あり)
- [x] 文節単体の候補の充実 (先頭の自立語を入れ替えた候補。「きょうは」に「京は」等が出る)
- [x] 文節区切りの手動調整 (Shift+←→ での伸縮)。縮小時は境界より後ろを固定せず
      自由に再変換し、伸縮の往復で元の区切りに戻るようにした
- [x] 記号辞書 (Mozc の symbol.tsv を読み込み、「やじるし」→「→」「ほし」→「★」等。
      通常語の後ろに追記する方式で、候補数上限には数えない)
- [x] 英単語入力 (生ローマ字候補)。打鍵列をそのまま半角候補にする (「apple」と打つと
      apple / Apple / APPLE)。かなに変換できない英字は composition に残し (MS-IME 同様)、
      英字を含む入力は全体を1文節に固定して変換する
- [x] ファンクションキー変換。F6-F10 は ひらがな/全角カタカナ/半角カタカナ/全角英数/半角英数
      への直接変換 (TSF層でローカルに文字種変換。変換前は全文、変換中は現在文節が対象)。
      F4 は ATOK 風の記号変換 (エンジンの CONVSYM で記号辞書の候補のみに絞る)
- [x] 記号キーと Shift+英字の composition 化。直接確定をやめ、未確定文字列として
      入力してから変換できるようにした (「（」を変換すると ［ 〔 等)。記号辞書の照合は
      読みを全角→半角に正規化して行う。JIS 配列の記号キーを一通りサポート。
      Shift+英字は英字モードに入り、以降の英字はローマ字変換せずアルファベットのまま続く
- [x] 対記号の両側同期変換。かっこ (（）｛｝「」) やクオート (”’) の対を含む入力で
      片側の候補を変えると、対になる側の文節も対応する記号に同期する
      (開き↔閉じは文節読みの深さ照合、左右同形のクオートは交互ペアで対応付け)
- [ ] 高速入力の工夫 (予測変換、省入力など)。2026-07-15 に既存IME・自作IMEを調査して
      優先度を整理した (詳細は下記「高速入力機能の検討結果」)

#### 高速入力機能の検討結果 (2026-07-15 調査)

Google日本語入力 / ATOK / macOS標準 / Mozc / azooKey / Akaza を調査し、費用対効果で優先度づけした。

**優先度A: 効果大・現構成に素直に乗る**

- [x] 履歴ベース予測変換 (2026-07-16 実装、フェーズ4-11)。learning.tsv の確定履歴を
      前方一致検索し (追記ログの行順から新しさを導出)、読み2文字以上で composition 下に
      サジェスト表示。Tab/↓で選択、↑で戻る (先頭でさらに↑は解除)、Enter で確定 (+LEARN)、
      Esc は選択解除のみの二段階。変換中の Tab は変換を取り消してサジェスト選択へ移行。
      英字モード・英字を含む読みでも履歴が前方一致すればサジェストする (英単語の省入力)。
      英字を含む入力は無変換の Enter 確定でも読み=表記で学習し、履歴に蓄積する
      (かなのみの無変換確定は、変換候補の並びを崩すため学習しない)。
      composition 中の Tab・矢印キー・PgUp/PgDn は用途が無くても常に食べ、
      アプリにキャレットやフォーカスを動かさせない。
      プロトコルに PREDICT コマンドを追加
- [x] 辞書ベース予測 (2026-07-16 実装、フェーズ4-11)。yada への辞書移行はせず、
      ソート済み読み配列 + 二分探索の前方一致検索で先行実装 (メモリ増は数十MB、
      yada 移行時に解消予定)。候補は履歴 (新しい順) → 辞書 (コスト順) の合成で最大8件。
      リアルタイム変換サジェストは未着手
      (Google日本語入力のサジェストは「入力履歴・システム辞書・リアルタイム変換」の3ソース合成。
      Mozc は src/prediction/ が独立サブシステムで、ダブル配列トライの前方一致検索が土台)
- [ ] 短縮よみ。フェーズ5のユーザ辞書登録に「短縮よみ」品詞を最初から入れる
      (ATOK の省入力データ相当。メールアドレスや定型文を数打鍵で出す)

**優先度B: 小さく作れて日々効く**

- [x] 日付・時刻の動的候補 (2026-07-15 実装)。「きょう」「あした (あす)」「あさって」
      「きのう」「おととい」「ことし」「らいねん」「きょねん」「いま」に現在日時の候補
      (2026/07/15、7月15日(水)、令和8年7月15日、14:30 など) を出す。
      エンジン側 (datetime.rs、chrono 使用)。当初は通常変換の候補に追記していたが、
      同音異義語の多い読みで候補が深く埋もれるため、F4 の特殊変換 (CONVSYM) に統合した
      (F4 で記号 → 日付の順に出る)。日付候補は学習から除外
      (翌日以降に古い日付が先頭に来るのを防ぐ)。数字表記変換 (「1234」→千二百三十四) は未着手
- [x] 確定アンドゥ (Ctrl+Backspace) (2026-07-15 実装)。直前の確定文字列と確定前の読み
      (コンポーザ) を覚えておき、composition が無いときの Ctrl+Backspace で確定文字列を
      削除して読みの composition に復元する。キャレット直前のテキストが確定文字列と
      一致するときのみ働き、一致しなければ何もしない (一度きりで記憶は消える)
- [ ] 再変換。確定済みテキストの選択範囲を再変換する (TSF の ITfFnReconversion 対応が必要でやや重い)

## 開発上の注意点

- **自分のマシンで直接デバッグしない**。IME の DLL は全アプリにロードされるため、バグるとテキスト入力全体が壊れたり、DLL がロードされたままで更新できなくなったりする
  - 開発初期はテスト専用の小さなホストアプリ (メモ帳で十分) で試す
  - Microsoft IME をいつでも切り替えられる状態を維持する
  - VM (Hyper-V) 上でテストできるとさらに安全
- TSF の公式ドキュメントは薄い。動いている実装 (CorvusSKK) を読むのが最良の資料
- 起動しっぱなしの古いエンジンプロセスが同名パイプへの接続を受けてしまい、新しいコードの
  テスト結果が古い挙動になることがある。エンジンのテスト前は `tasklist | grep quicklime` で
  旧プロセスの有無を確認して終了しておく
- Windows 11 のメモ帳はウィンドウを閉じてもプロセスが残り、新しいウィンドウも同じプロセス
  (= 古い DLL をロードしたまま) で開く。TSF 層のテスト前は `taskkill /IM Notepad.exe /F` で
  プロセスごと終了してから開き直す
- 再ビルド時に DLL がプロセスにロックされてリンクに失敗する場合は、ロード中でもリネームは可能なので
  `mv QuicklIME.dll QuicklIME.dll.old` と退避してからリンクし直す。古い DLL を掴んだプロセスは
  再起動するまで旧版のまま動く点に注意 (テストは新しく起動したアプリで行う)
